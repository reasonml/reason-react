Creación, Props y Self

La documentación asume cierta familiaridad con ReactJS.

ReasonReact no usa -ni necesita- classes. La API de creación de componentes devuelve un record cuyos atributos (como render por ejemplo) se pueden sobreescribir.

Un template the componente es creado llamando a la siguiente función: ReasonReact.statelessComponent("TheComponentName"). The string being passed is for debugging purposes (the equivalent of ReactJS' displayName).

Como ejemplo, aquí está el archivo Greeting.re:

let component = ReasonReact.statelessComponent("Greeting");

En ReactJS, crearíamos un componente heredando de Component y lo llamaríamos usando JSX el cuál se transformaría en React.createElement(myClass, {prop1: 'hello'}). En ReasonReact, en vez de pasar la clase (aka el template del componente) a la hipotética función ReasonReact.createElement declaramos la función make:

/* still in Greeting.re */
let component = ReasonReact.statelessComponent("Greeting");

let make = (~name, _children) => {
  ...component, /* expande los valores por defecto de component aquí  */
  render: _self => <div> {ReasonReact.stringToElement(name)} </div>
};

La función make es llamada por la implementación de JSX de ReasonReact, la cual cubriremos más adelante. Por ahora, la forma sin JSX de llamar y renderear un componente es la siguiente:

ReasonReact.element(Greeting.make(~name="John", [||])) /* the `make` function in the module `Greeting` */
/* equivalent to <Greeting name="John" /> */

La función make debe devolver un record de tipo componente como el creado arriba. Al record se le pueden sobreescribir atributos como el conocido render, initialState, didMount, etc. De todos estos hablaremos luego.

Nota: no declare la expresión let component dentro de la función make como en el siguiente ejemplo!

let make = _children => {
  ...(ReasonReact.statelessComponent("Greeting")),
  render: self => blabla
}

Ya que la función make es llamada con cada invocación JSX, de hacerlo estaríamos creando un nuevo componente cada vez.

Props

Las props son simplemente los labeled arguments (argumentos referenciados por nombre, no por posición) de la función make que vimos anteriormente. Los mismos pueden ser opcionales y/o tener valores por defecto, e.j. let make = (~name, ~age=?, ~className="box", _children) => ....

La última prop siempre debe ser children. En caso de no usarla, se puede ignorar nombrándola _ o _children. Los nombres que empiezan con infraguión no disparan errores de compilación si no son usados.

Asimismo no se pueden usar props con nombre ref ni key. Al igual que en ReactJS estos nombres son reservados.

Siguiendo con el ejemplo se podría llamar al componente desde otro archivo del siguiente modo: <Foo name="Reason" />. En este caso className que fue omitido por defecto el valor "box" y age el valor None. Si quisiéramos pasarle explícitamente un valor a age, podemos hacerlo de la siguiente manera: <Foo name="Reason" age=20 />.

Un Truco Elegante para hacer Props Forwarding

Muchas veces en ReactJS recibimos props que queremos pasar directamente a algún componente hijo:

<Foo name="Reason" age={this.props.age} />

Esto es una fuente de errores ya que this.props.age puede ser accidentalemte cambiado a un Number que acepte valores null mientras que Foo no lo espera así, o vice-verse, podríamos estar trabajando anteriormente con un valor nullable y cambiarlo para de esta forma dejar a Foo con una validación inecesaria en algún lugar de su método render.

En Reason si queremos explicitamente pasar ageFromProps de manera opcional (cuyo tipo es option int, aka None | Some int), el siguiente código no funcionaría:

<Foo name="Reason" age=ageFromProps />

Esto sucede porque age espera un int cuando es llamado desde Foo no un option int! Ingenuamente podríamos resolver el problema de la siguiente manera:

switch (ageFromProps) {
| None => <Foo name="Reason" />
| Some(nonNullableAge) => <Foo name="Reason" age=nonNullableAge />
}

Demasiado engorroso. Afortunadamente hay una mejor forma de pasar explícitamente un valor opcional:

<Foo name="Reason" age=?ageFromProps />

Lo que se traduce a: "Entiendo que age es opcional y que, cuando uso el label debo pasar un int. Pero voy a hacerlo pasando un valor option explícitamente". Esto no es un truco que nos hayamos inventado; es una característica del lenguaje! Lee la sección acerca de "Pasando Optional Explícitamente" en la documentación de Reason.

self

Quizás hayas notado el siguiente código render: (self) => ... que es parte de la función make. El concepto this como lo conocemos en JavaScript no existe en ReasonReact (pero puede existir en Reason debido a su sistema opcional de objetos); el equivalente a this es self. Es un record que contiene state, retainedProps, handle y reduce, el cual pasamos a los eventos de ciclo de vida, render y algunos otros, cuando necesitan un "saco de información". Estos conceptos se explicarán más adelante.
